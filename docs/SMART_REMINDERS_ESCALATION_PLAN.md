# План: умные напоминания и авто-эскалация (продакшен)

## Текущая реализация (кратко)

### Автонапоминания
- **Файл:** `lib/reminders/run-auto-reminders.ts`
- **Cron:** `GET /api/cron/reminders` (защита `CRON_SECRET`), вызывается раз в день (например 9:00 UTC).
- **Логика:** Выбираются инвойсы с `auto_remind_enabled = true`, `client_email` не пустой, `sent_at` не null, статус один из `sent` / `viewed` / `overdue`. Для каждого инвойса по расписанию `auto_remind_days` (например `1,3,7`) считается, прошло ли N дней после `sent_at`; если да и соответствующее `reminder_Nd_sent_at` ещё null — отправляется письмо-напоминание (Resend), обновляются `reminder_Nd_sent_at` и `last_reminder_at`.
- **Шаблон:** `lib/email/send.ts` → `sendReminderEmail`, шаблон в `lib/email/templates.ts` (buildReminderEmailHtml). Тема: "Reminder: Invoice … from …".
- **Ограничения:** Нет копии владельцу, нет эскалации (один и тот же текст на 1/3/7 день), нет привязки к `due_date`.

### Ручное напоминание
- **Действие:** `sendReminderAction(invoiceId)` в `app/invoices/actions.ts`.
- **Лимит:** `REMINDER_RATE_LIMIT_HOURS` (по умолчанию 24) — не чаще одного напоминания в N часов на инвойс (`last_reminder_at`).
- Копии владельцу нет.

### Статус Overdue
- В БД статус `overdue` есть в enum, но **ни один cron/сервис не проставляет его**. В дашборде и аналитике overdue считается «на лету»: неоплаченный инвойс с `due_date < today` учитывается как overdue в счётчиках и суммах (`lib/dashboard/stats.ts`, `lib/dashboard/analytics.ts`).
- В списке инвойсов фильтр «Overdue» опирается на **сохранённый** `status === 'overdue'` (`invoice-list.tsx`), поэтому без отдельного джоба, выставляющего статус, фильтр по overdue может не совпадать с реальностью.

### Отслеживание платежей
- Оплата: webhook Stripe обновляет `paid_at` и `status = 'paid'`.
- Ручное «Mark as paid» выставляет то же через `markAsPaidManualAction`.
- Отдельного «трекинга платежей» в виде отдельного модуля нет — всё через статусы инвойса и поля `sent_at` / `viewed_at` / `paid_at`.

## Цель доработки

Довести функцию до «крутого продакшен-состояния»:

1. **Умные напоминания:** не только «через N дней после отправки», но и привязка к сроку оплаты и эскалация (разный тон/частота).
2. **Авто-эскалация:** например, после 7 дней — второе напоминание с копией тебе; после 14 дней — пометить overdue и уведомить владельца.
3. **Единый cron:** один джоб, который и проставляет overdue, и рассылает напоминания по правилам.
4. **Копия владельцу:** опция «копия мне на почту» при напоминаниях (и при эскалации).
5. **Прозрачность:** активность (activity) и, при желании, простой лог «что отправили и когда».

## План работ (по шагам)

### Фаза 1: Стабильная основа (overdue + один источник правды)

1. **Cron: проставление статуса Overdue**
   - Новый джоб (или шаг в существующем cron напоминаний): раз в день обновлять инвойсы:
     - `status IN ('sent', 'viewed')` и `due_date IS NOT NULL` и `due_date < today` → `status = 'overdue'`.
   - Файл: либо расширить `lib/reminders/run-auto-reminders.ts`, либо вынести в `lib/reminders/set-overdue-status.ts` и вызывать из того же `/api/cron/reminders` до запуска напоминаний.
   - Миграции не требуются (поле и enum уже есть).

2. **Единая точка входа cron**
   - В `GET /api/cron/reminders`: сначала вызвать «set overdue», затем `runAutoReminders()`.
   - В ответе JSON можно добавить поля `overdueUpdated: number` и оставить `sent`, `errors` для напоминаний.

Итог: фильтр «Overdue» в списке инвойсов и отчёты по overdue будут совпадать с реальностью; последующие правила эскалации могут опираться на `status = 'overdue'`.

### Фаза 2: Правила напоминаний (от due_date и эскалация)

3. **Модель правил (конфиг в коде или БД)**
   - Вариант A (простой): завести в коде константы/массив правил, например:
     - «Напоминание 1»: через 1 день после `sent_at` (как сейчас).
     - «Напоминание 2»: через 3 дня после `sent_at` или в день `due_date` (если due_date задан).
     - «Напоминание 3»: через 7 дней после `sent_at` или через 3 дня после `due_date`.
     - «Эскалация»: если статус уже `overdue` и прошло ≥ 7 дней с `due_date` — отправить «второе напоминание» клиенту + копия владельцу и/или пометить «эскалация отправлена».
   - Вариант B (гибкий): таблица `reminder_rules` (user_id, trigger_type: 'after_sent' | 'after_due', trigger_days, send_owner_copy, template_key) и выборка по user_id. Для первой итерации достаточно Варианта A.

4. **Расширение run-auto-reminders**
   - Учитывать `due_date` при расчёте «когда отправить»:
     - Например: «напоминание в день due_date» (если due_date в прошлом или сегодня — отправить один раз).
     - Или: «напоминание через N дней после due_date» (для overdue).
   - Ввести понятие «эскалация»: после X дней в статусе overdue отправить усиленное напоминание и копию владельцу (см. ниже).
   - Оставить обратную совместимость: если `due_date` нет, поведение как сейчас (только по `sent_at`).

5. **Копия владельцу**
   - При отправке напоминания (и эскалации) опционально отправлять копию на email владельца (из `profiles` или `auth.users`).
   - Параметр: глобальный (настройка в Settings: «Копия мне на все напоминания») или только для «эскалации». Рекомендация: сначала только для эскалации, потом общая настройка.
   - В Resend: второй вызов `emails.send` на email владельца с темой вида «[Puyer] Reminder sent: Invoice #X to Client» и кратким текстом (ссылка на инвойс, сумма, клиент).

6. **Шаблоны писем**
   - Текущий reminder — один шаблон. Для эскалации можно:
     - Вариант A: тот же шаблон, тема «Second reminder: Invoice …» или «Reminder (overdue): Invoice …».
     - Вариант B: отдельный HTML (buildEscalationEmailHtml) с более жёстким формулировками («Payment is overdue», «Please pay by …»).
   - В плане достаточно одного шаблона с разными subject/префиксами; отдельный шаблон — по желанию.

7. **Хранение факта эскалации**
   - Добавить в таблицу `invoices` колонку `escalation_sent_at` (timestamptz, nullable). После отправки «эскалации» (второе напоминание с копией владельцу) выставлять её и больше не слать эскалацию по этому инвойсу.
   - Миграция: `add column if not exists escalation_sent_at timestamptz`.

### Фаза 3: Настройки в UI и прозрачность

8. **Настройки напоминаний (Settings)**

   - Опционально: «Считать overdue через N дней после due_date» (сейчас 0 — в день due_date). Пока можно зашить 0 в коде.

### Фаза 4: Надёжность и мониторинг

11. **Идемпотентность и лимиты**
    - Один и тот же тип напоминания (например 7d) не отправлять дважды (уже обеспечено полями `reminder_Xd_sent_at`).
    - Эскалацию отправлять один раз (`escalation_sent_at`).
    - Сохранить лимит на ручные напоминания (`REMINDER_RATE_LIMIT_HOURS`); для авто-напоминаний лимит по сути «раз в день на тип» (1d, 3d, 7d…).

12. **Unsubscribe**
    - Напоминания уже проверяют `email_unsubscribes` через `isEmailUnsubscribed` в `sendReminderEmail`. Сохранить это для всех новых типов писем (reminder, escalation).

13. **Логирование**
    - В `runAutoReminders` логировать количество отправленных напоминаний и ошибок (уже есть). Добавить краткий лог при установке overdue и при отправке эскалации (без PII: invoice id, type).
    - Опционально: метрики (сколько overdue обновлено, сколько эскалаций отправлено) для будущего дашборда админа.

14. **Тесты**
    - Модульный тест: расчёт «когда отправить» для заданных `sent_at`, `due_date`, правил (день 1, 3, 7, эскалация через 7 дней после due).
    - Интеграционный: мок Resend, вызов runAutoReminders и проверка, что нужные инвойсы получили обновление полей и вызовы send.

## Риски и ограничения

- **Resend лимиты:** увеличение числа писем (клиент + владелец) укладывать в квоту; при массовом overdue — возможен burst. Можно добавить простой throttle (например не более N писем в минуту в runAutoReminders).
- **Спам:** текст эскалации не должен быть агрессивным; наличие unsubscribe и соблюдение лимитов снижают риск жалоб.
- **Часовые пояса:** сейчас расчёт «день» по UTC. Для «в день due_date» при необходимости можно учитывать timezone профиля (поле уже есть в onboarding).
